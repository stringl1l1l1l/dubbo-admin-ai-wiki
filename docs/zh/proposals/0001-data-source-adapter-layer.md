# 0001 数据源适配层

## 状态

已采纳

## 1. 为什么需要这个提案

当前 RAG 流程最终要处理的是“标准化文档”，但真实世界的数据来源并不统一：本地文件、HTTP、对象存储、数据库，都会有不同的读取、清洗和元信息结构。

如果上层索引流程直接依赖每种数据源的读取逻辑，就会出现几个问题：

- 新接一个数据源就要复制一遍处理链路
- 不同数据源输出格式不一致
- 测试只能靠端到端验证，难以针对读取层单独验证
- 数据接入能力和 RAG 主流程耦合过深

因此，这个提案要解决的不是“如何多支持几个数据源”，而是“如何把数据读取从索引主流程中解耦出来”。

## 2. 目标

- 为不同数据源提供统一读取抽象
- 让索引流程只面向标准化文档流工作
- 降低新增数据源的接入成本
- 提升测试性和可观测性

## 3. 非目标

- 不在本提案中重写检索逻辑
- 不在本提案中替换现有向量后端
- 不改变对外 HTTP API

## 4. 设计概览

引入 `DataSourceAdapter`，负责把原始数据源转换为统一的文档读取接口：

```go
type DataSourceAdapter interface {
    Name() string
    Validate(cfg map[string]any) error
    Open(ctx context.Context, cfg map[string]any) (DocumentReader, error)
}

type DocumentReader interface {
    Next(ctx context.Context) (*Document, error)
    Close() error
}
```

这层抽象的重点不是炫技，而是把“读取”和“后处理”明确分层：

- 适配层：负责读取和标准化
- 后处理层：负责清洗、切分、去重、索引

## 5. 目标架构

```mermaid
flowchart LR
  Source["文件 / HTTP / DB / OSS"] --> Adapter["DataSourceAdapter"]
  Adapter --> Reader["DocumentReader"]
  Reader --> Normalize["标准化 Document"]
  Normalize --> Pipeline["清洗 / 切分 / 去重 / 索引"]
```

## 6. 标准化文档模型建议

建议统一保留以下核心字段：

- `id`
- `title`
- `content`
- `source`
- `updated_at`
- `tags`

扩展字段进入 `metadata`，避免主模型膨胀过快。

## 7. 为什么这个设计合理

### 对 RAG 主流程更友好

上层只需要关心“拿到文档后怎么处理”，不需要关心“文档原来从哪里来”。

### 对测试更友好

可以为每个数据源单独写适配器测试，也可以对标准化输出写统一测试。

### 对扩展更友好

新增数据源时，只要实现适配器，不必重写整个索引链路。

## 8. 风险

### 风险一：抽象过度

过度追求统一可能牺牲某些数据源的特性表达能力。

应对方式：

- 核心字段收敛
- 特有字段进入 `metadata`

### 风险二：性能损耗

如果抽象只支持“全量加载”，在大数据源下会非常低效。

应对方式：

- 设计为 `DocumentReader` 流式读取
- 为批量场景预留扩展空间

### 风险三：迁移期行为不一致

旧逻辑和新适配层可能对同一数据源产生不同输出。

应对方式：

- 对关键数据集做回归对比
- 分批迁移数据源

## 9. 迁移建议

1. 先实现一个本地文件适配器作为基线。
2. 把现有索引流程改为消费 `DocumentReader`。
3. 对旧逻辑和新逻辑做结果对比。
4. 再逐步接入其他数据源。

## 10. 预期收益

- 数据接入能力更清晰
- RAG 主流程更稳定
- 测试粒度更细
- 文档和代码的模块边界更清楚
